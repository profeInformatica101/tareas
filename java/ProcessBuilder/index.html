public class Utils {
    public static String Consola() {
        String linea = null;
        try {
            BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
            while (true) {

Desarrolla un programa que consulte una imagen aleatoria de la API pública https://dog.ceo/api/breeds/image/random usando un proceso externo con curl. El programa debe leer la respuesta JSON, extraer la URL de la imagen, mostrarla por consola y abrirla en el navegador predeterminado del sistema. Además, mostrará el PID del proceso, medirá la duración de la petición y comprobará si el proceso sigue vivo para finalizarlo si fuese necesario.

Requisitos:
  • Usar ProcessBuilder para lanzar curl en modo silencioso.
  • Unificar salida y error con redirectErrorStream(true) para simplificar la lectura.
  • Leer la salida JSON de la API y extraer la URL del campo message.
  • Medir el tiempo de ejecución con Instant y Duration.
  • Mostrar PID, duración y el código de finalización del proceso.
  • Abrir la URL resultante en el navegador predeterminado (Windows/macOS/Linux).
  • Comprobar si el proceso sigue activo y destruirlo en caso necesario.
Ejemplo de salida esperada:
🐶 Consultando imagen aleatoria (PID: 7352)...
⏱️ Duración: 187 ms
🌐 Imagen: https://images.dog.ceo/breeds/husky/n02110185_1469.jpg
🖥️ Abriendo imagen en el navegador...
✅ Proceso finalizado correctamente.

Se recomiendan los siguientes métodos auxiliares para procesar el JSON y abrir la imagen: 

// Extrae la URL entre comillas del campo "message"
private static String extraerURL(String json) {
    int start = json.indexOf("\"message\":\"");
    if (start < 0) return null;
    start += 11;
    int end = json.indexOf("\"", start);
    if (end < 0) return null;
    String raw = json.substring(start, end);
    return raw.replace("\\/", "/"); // limpiar los escapes
}

// Abre la URL en el navegador según el sistema operativo
private static void abrirEnNavegador(String imageUrl) {
    String os = System.getProperty("os.name").toLowerCase();
    ProcessBuilder abrir;
    try {
        if (os.contains("win")) {
            abrir = new ProcessBuilder("cmd", "/c", "start", imageUrl);
        } else if (os.contains("mac")) {
            abrir = new ProcessBuilder("open", imageUrl);
        } else {
            abrir = new ProcessBuilder("xdg-open", imageUrl);
        }
        abrir.start();
        System.out.println("🖥️  Abriendo imagen en el navegador...");
    } catch (IOException e) {
        System.err.println("No se pudo abrir la imagen: " + e.getMessage());
    }
}

- Construye el proceso con curl hacia la URL indicada y activa el modo silencioso.
- Unifica stdout y stderr para leer la salida con un único flujo.
- Captura el PID mediante la API de procesos y registra el instante de inicio.
- Lee la salida completa del proceso; al finalizar, calcula la duración transcurrida.
- Usa el método extraerURL() para obtener la URL de la imagen desde el JSON.
- Llama a abrirEnNavegador() para abrir la imagen automáticamente.
- Si el proceso continuara activo al final, ciérralo de forma controlada y registra el estado.
- Muestra mensajes claros y ordenados para cada paso (inicio, duración, URL, apertura, fin).
- No es necesario utilizar librerías JSON externas: basta con analizar la cadena manualmente.

Crea un programa que construya un informe atractivo del sistema combinando varios procesos del SO y, al menos en un bloque, un pipeline de comandos. El informe debe mostrar: versión del kernel, distribución, CPU, memoria, discos, tiempo activo, IP(es) y ubicación aproximada de la IP pública.

Fuentes sugeridas (Linux):
  • 🧬 Kernel: uname -srmo
  • 🎛️ Distro: lsb_release -ds o lectura de /etc/os-release con grep
  • 🧠 CPU: lscpu filtrado con grep/awk
  • 💾 Memoria: free -h
  • 💿 Discos: df -h
  • ⏱️ Uptime: uptime -p y uptime -s
  • 🌐 IP privada(s): ip -br a (o hostname -I)
  • 🛰️ IP pública: curl -s https://api.ipify.org?format=json
  • 🗺️ Geo: curl -s https://ipinfo.io/json y extrae ciudad o país.

Puedes lanzar varios procesos en paralelo para acelerar la recopilación de información. Este es el patrón básico recomendado: 

// Ejemplo de patrón para lanzar y esperar varios procesos
ProcessBuilder pb1 = new ProcessBuilder("uname", "-srmo");
ProcessBuilder pb2 = new ProcessBuilder("free", "-h");
ProcessBuilder pb3 = new ProcessBuilder("curl", "-s", "https://api.ipify.org?format=json");

List<Process> procesos = List.of(pb1.start(), pb2.start(), pb3.start());

// Esperar todos los procesos
for (Process p : procesos) {
    p.waitFor();
}

// Leer salidas
for (Process p : procesos) {
    try (var r = new java.io.BufferedReader(
            new java.io.InputStreamReader(p.getInputStream()))) {
        r.lines().forEach(System.out::println);
    }
}

- Construye cada sección del informe con ProcessBuilder y formatea la salida.
- Implementa al menos un bloque con pipeline (por ejemplo, lscpu | grep).
- Usa Instant y Duration para medir el tiempo total.
- No mezcles errores con salida estándar: usa getErrorStream() para diagnóstico.
- Cierra los flujos de salida de los procesos intermedios para evitar bloqueos.
- Presenta el informe con emojis o iconos para hacerlo visualmente atractivo.
- Controla los casos sin conexión a Internet y muestra un mensaje claro.
- Espera a todos los procesos lanzados antes de finalizar el programa.
- Asegúrate de limpiar bien los InputStream y destruir procesos que sigan activos.

Diseña un programa (o script lanzado desde Java con ProcessBuilder) que automatice el flujo básico de subida a GitHub desde un repositorio local: crear una rama nueva, añadir todos los cambios, hacer un commit descriptivo y subir la rama al remoto. El foco es **automatizar con robustez**: detectar y manejar errores comunes (estado sucio, conflictos, fallos de red, autenticación), informar al usuario y garantizar que **solo** se suban cambios válidos.

Requisitos mínimos:
  • Crear una rama nueva con un nombre seguro (p. ej. feat/tu-nombre-descriptivo), asegurando que no exista previamente en remoto.
  • Comprobar el estado del repositorio (sin merges pendientes, sin índices bloqueados). Si hay cambios no añadidos, informarlo y permitir opción de continuar o abortar.
  • Ejecutar git add . y comprobar su exit code antes de proceder.
  • Generar un commit con un mensaje claro; si no hay cambios a commitear, abortar con aviso (no empujar ramas vacías).
  • Configurar la rama remota/upstream si es la primera vez (p. ej. git push --set-upstream origin <branch>), manejando la salida y errores.
  • Detectar errores comunes al hacer push: autenticación, permisos, divergencia (need pull/merge), tiempos de espera de red. Mostrar mensajes útiles y recomendaciones (pull/rebase, revisar conflictos, comprobar credenciales).
  • Evitar operaciones destructivas automáticas (no usar git push --force automáticamente). Si se propone forzar, pedir confirmación explícita y documentarla en el README del ejercicio.
  • Registrar en consola (y en log opcional) cada paso con PID del proceso, comando ejecutado, exit code y salida estándar/errores breves para diagnóstico.

- Antes de crear la rama, comprueba que estás en el repositorio correcto y que el directorio de trabajo no está en un estado inconsistente (git status --porcelain).
- Genera un nombre de rama seguro y único (incluye prefijo tipo feat/ o fix/ y una parte descriptiva).
- Comprueba si la rama ya existe en remoto (git ls-remote --heads origin <branch>) para evitar colisiones.
- Ejecuta los comandos secuencialmente como procesos hijos y **comprueba el código de salida** de cada uno; si uno falla, muestra la salida de error y detén la secuencia.
- Antes del push, valida que existe al menos un commit nuevo en la rama (si no, no empujar).
- Para el push inicial usa la opción para crear upstream y captura mensajes que pidan autenticación; instruye sobre el uso de SSH keys o tokens personales (no guardar tokens en texto plano).
- Implementa reintentos simples con backoff en caso de fallos transitorios de red (p. ej. 3 intentos con espera creciente).
- Maneja la divergencia: si el push falla por commits remotos no integrados, muestra instrucciones claras para realizar git pull --rebase o resolver conflictos manualmente; no intentes resolver merges automáticamente sin aviso.
- Añade un modo "dry-run" (ejecutar git push --dry-run) para que la herramienta muestre lo que haría sin modificar el remoto, útil para pruebas en clase.
- En cuestiones de autenticación, sugiere usar: claves SSH bien configuradas, o el helper de credenciales de Git con un token (y recuerda no almacenar tokens en repositorios ni en el código).
- Incluye mensajes finales claros: URL del repositorio remoto + rama subida, o instrucciones de error y próximos pasos si no se pudo completar la subida.
Precauciones de seguridad: nunca codifiques credenciales en el programa. Para la práctica, recomienda usar claves SSH con passphrase o tokens almacenados en el gestor de credenciales del sistema.

Crea un script util-simple.sh que demuestre cómo se controlan los parámetros y opciones en Bash, y ejecútalo desde Java con ProcessBuilder. El script deberá:
  • Recibir un nombre como primer argumento y mostrar “Hola, <nombre>”.
  • Aceptar --upper para mostrar el saludo en MAYÚSCULAS.
  • Aceptar --count <carpeta> para mostrar cuántos ficheros contiene esa carpeta.
  • Si no se pasa nombre, leerlo desde stdin (permite echo Ana | ./util-simple.sh --upper).
  • Usar códigos de salida:
    • 0 → todo correcto
    • 1 → argumentos inválidos
    • 2 → carpeta inexistente

- Crea el script con cabecera #!/usr/bin/env bash y marca permisos de ejecución (chmod +x util-simple.sh).

Control de parámetros en Bash:
En los scripts Bash, los parámetros se controlan mediante variables automáticas:
  • $0 → nombre del script
  • $1, $2, $3... → argumentos posicionales
  • $# → número de argumentos pasados
  • $@ → todos los argumentos (como lista)
  • $* → todos los argumentos en una sola cadena
  • shift → avanza los argumentos (útil para recorrerlos en bucles)
Ejemplo básico de control manual: 
while [[ $# -gt 0 ]]; do
  case "$1" in
    --upper) MAYUS=1 ;;
    --count) CARPETA="$2"; shift ;;
    *) NOMBRE="$1" ;;
  esac
  shift
done
- Valida si $NOMBRE está vacío; si lo está, léelo desde stdin con read NOMBRE.
- Si la opción --count está activa, usa ls -1 "$CARPETA" | wc -l para contar ficheros.
- En modo --upper, transforma la cadena con echo "$NOMBRE" | tr '[:lower:]' '[:upper:]'.
- Devuelve el código de salida adecuado con exit 0, exit 1 o exit 2 según corresponda.

Integración desde Java:
- Lanza el script con new ProcessBuilder("./util-simple.sh","--upper","Pepe") o con redirección de entrada (echo Ana | ./util-simple.sh).
- Captura stdout y stderr para mostrar resultados y errores.
- Muestra el exit code con process.waitFor() y explica su significado.
- Usa ProcessBuilder.Redirect.INHERIT si quieres que el script escriba directamente en la consola de Java.

Crea un script util-simple.sh dentro de la carpeta /home/<usuario>/Documents/scripts/ y ejecútalo desde Java utilizando ProcessBuilder. Este ejercicio combina el uso de parámetros y variables de entorno que el script debe leer y mostrar.

Objetivos del script:
  • El script debe ubicarse en /home/<usuario>/Documents/scripts/util-simple.sh.
  • Recibirá un nombre como primer argumento y mostrará “Hola, <nombre>”.
  • Aceptará las opciones:
    • --upper → muestra el saludo en mayúsculas.
    • --count <carpeta> → muestra cuántos ficheros contiene la carpeta indicada.
  • Si no se pasa nombre, lo leerá desde stdin (por ejemplo: echo Ana | ./util-simple.sh).
  • Usará variables de entorno que el programa Java le proporcionará:
    • APP_MODE (por ejemplo: “DEV” o “PROD”)
    • AUTHOR (nombre del alumno o creador)
    El script debe leerlas y mostrarlas al inicio de su ejecución.
Códigos de salida esperados:
  • 0 → todo correcto
  • 1 → argumentos inválidos
  • 2 → carpeta inexistente

El propio util-simple.sh será responsable de determinar y devolver estos códigos:

  • El código 1 se devuelve si el usuario no pasa los argumentos esperados o usa una opción incorrecta. El script puede detectarlo comprobando el número de argumentos ($#) o validando las opciones con un bucle case.
  • El código 2 se devuelve si la carpeta indicada en --count no existe. Para ello se usa la comprobación [[ -d "$CARPETA" ]]; si falla, se muestra un mensaje de error por stderr y se hace exit 2.
  • El código 0 se devuelve cuando todo se ejecuta correctamente (saludo y conteo opcional) sin errores.

1. Ubicación y permisos
- Crea la carpeta si no existe: mkdir -p /home/<usuario>/Documents/scripts
- Crea el script en esa ruta: nano /home/<usuario>/Documents/scripts/util-simple.sh

2. Control de parámetros en Bash
Bash permite leer parámetros mediante variables especiales:
  • $1, $2, $3... → argumentos posicionales
  • $# → número de argumentos
  • $@ → todos los argumentos como lista
  • shift → avanza al siguiente argumento
Ejemplo de análisis con case: 
while [[ $# -gt 0 ]]; do
  case "$1" in
    --upper) MAYUS=1 ;;
    --count) CARPETA="$2"; shift ;;
    *) NOMBRE="$1" ;;
  esac
  shift
done

3. Variables de entorno
En el script, las variables de entorno enviadas desde Java se leen igual que cualquier otra: 
echo "Modo: $APP_MODE"
echo "Autor: $AUTHOR"
Si alguna no está definida, usa un valor por defecto: 
APP_MODE="${APP_MODE:-DEV}"

4. Ejecución desde Java (ProcessBuilder)

- Usa variables de entorno: pb.environment().put("APP_MODE", "PROD");

- Captura stdout y stderr para mostrar los mensajes del script.
- Muestra el PID y el código de salida con process.toHandle().pid() y process.waitFor().
- Si hay error (código ≠ 0), muestra un mensaje según el valor devuelto.

5. Buenas prácticas
- Usa set -e en el script para detenerlo si hay errores graves.
- Comprueba si la carpeta de --count existe con [[ -d "$CARPETA" ]] || exit 2.
- Devuelve un código de salida con exit y explica su significado.
